1. Mean Rank (R-M)
Intuition: Mean Rank treats each evaluation (e.g. each dataset or task in a benchmark) as giving each algorithm a position (1st, 2nd, 3rd, etc.), and then simply averages those positions. In other words, it asks "on average, what rank does this algorithm achieve?" across all tasks. This is analogous to taking an athlete's average finishing place over many races.
Technical details: Formally, if an algorithm iii has rank ri,jr_{i,j}ri,j​ on task jjj (where rank 1 means best performance), the mean rank is R-M(i)=1m∑j=1mri,jR\text{-}M(i) = \frac{1}{m}\sum_{j=1}^{m} r_{i,j}R-M(i)=m1​∑j=1m​ri,j​, where mmm is the number of tasks. A lower mean rank indicates better overall performance. This aggregation assumes each task is equally important and focuses purely on relative ordering, not the magnitude of performance differences. It's straightforward and often used in statistical tests (like Friedman tests) as a summary of performancear5iv.labs.arxiv.org. However, it can be sensitive to outliers: an algorithm that performs very poorly (high rank) on one task can have its mean rank pulled up significantly, even if it is top-ranked on others. Mean Rank is useful when you want a simple, overall ranking and when all tasks are considered of equal weight in terms of ranking.
2. Median Rank (R-Md)
Intuition: Median Rank looks at the typical rank an algorithm gets, rather than the average. Imagine observing an algorithm's ranks across tasks and asking "what rank does it usually achieve?". The median rank answers that by taking the middle value of the ranks, which dampens the effect of any one extreme result. It's like saying, "half the time this method is at or above rank X."
Technical details: Formally, if an algorithm has ranks {ri,1,ri,2,…,ri,m\{r_{i,1}, r_{i,2}, \dots, r_{i,m}{ri,1​,ri,2​,…,ri,m​} across mmm tasks, the median rank R-Md(i)R\text{-}Md(i)R-Md(i) is the median of this set of ranks. If mmm is odd, this is the middle-ranked value when sorting the ranks; if mmm is even, it can be the average of the two middle ranks or chosen by a convention. The median rank is more robust to outliers than the mean rank - a single very bad (or very good) performance won't sway it as much. This makes it useful when performance varies widely: for example, if an algorithm is usually among the top performers but has one or two disastrous results, the median rank will still reflect that it's typically near the top. On the downside, median rank can conceal consistent small differences (two algorithms with the same median rank might have different distributions of ranks). It's best used when we care about an algorithm's typical performance and want to minimize the influence of anomalies.
3. Mean Quality (Q-M)
Intuition: Mean Quality looks at the actual performance scores (often called "qualities") of an algorithm on each task and averages them. Instead of looking at positions, we ask "what is the algorithm's average score across tasks?". For example, if the metric is accuracy, Q-M is the average accuracy over all datasets, giving a direct sense of overall performance level.
Technical details: Let Qi,jQ_{i,j}Qi,j​ be the performance metric (quality) of algorithm iii on task jjj - for instance, accuracy, F1 score, or error rate (with a suitable sign convention). The mean quality is Q-M(i)=1m∑j=1mQi,jQ\text{-}M(i) = \frac{1}{m}\sum_{j=1}^{m} Q_{i,j}Q-M(i)=m1​∑j=1m​Qi,j​. This aggregation assumes the performance metrics are comparable across tasks (e.g. all are accuracies or all are measured on similar scales). A higher Q-M indicates better overall performance if the metric is something to maximize (like accuracy). Mean Quality captures magnitude of performance differences: an algorithm that is slightly better on every task will have a higher Q-M than one that is sometimes best and sometimes worst, even if their ranks vary. One must be cautious, however, because tasks with inherently higher scores can dominate the average - each task is weighted proportionally to its absolute metric values. In practice, this means that if one dataset yields generally higher accuracies for all algorithms (e.g. an easy dataset), it contributes more to the mean than a harder dataset with lower scores, unless metrics are normalized. Despite this, Q-M is useful for an overall sense of how an algorithm might perform quantitatively on a random task. It's straightforward to compute, but unlike rank-based methods it does not inherently account for variability or statistical significance of differences.
4. Median Quality (Q-Md)
Intuition: Median Quality focuses on the typical score an algorithm achieves, rather than the average score. It asks "what performance can this algorithm achieve at least half the time?". By taking the median of the scores across tasks, we get a sense of the algorithm's central performance tendency, ignoring extreme highs or lows.
Technical details: Given performance values {Qi,1,Qi,2,…,Qi,m}\{Q_{i,1}, Q_{i,2}, \dots, Q_{i,m}\}{Qi,1​,Qi,2​,…,Qi,m​} for algorithm iii across mmm tasks, the median quality Q-Md(i)Q\text{-}Md(i)Q-Md(i) is the median of these values. If the number of tasks is odd, this is the middle value when scores are sorted; if even, it can be the average of the two middle values or defined by convention. Median Quality is a robust aggregator, less sensitive to outlier tasks than the mean. For instance, if an algorithm's scores are mostly around 0.8 but one task was an outlier with 0.4, the median will stay around 0.8 whereas the mean would be pulled down. This makes Q-Md useful when we suspect some tasks are not representative or have high variance in difficulty - it tells us the performance level the algorithm achieves in a typical scenario. However, like any median, it discards information about the tails: an algorithm that occasionally has extremely high scores won't get credit for those in the median. Thus, median quality is ideal when consistency is valued over occasional brilliance (or when we want to mitigate the impact of occasional failures). It provides a "safety belt" view of performance - how good an algorithm is in general, most of the time.
5. Rescaled Mean Quality (Q-RM)
Intuition: Rescaled Mean Quality first normalizes performance on each task before averaging, to ensure each task contributes fairly. Think of it as grading on a curve for each dataset: an algorithm's score on a task is converted into a 0–1 scale based on the worst and best performance achieved on that task. Then those normalized scores are averaged. This answers "on average, how close to the best performance is this algorithm, across tasks?" without letting tasks with larger raw scores dominate the averagear5iv.labs.arxiv.org.
Technical details: For each task jjj, let Qjmin⁡Q_{j}^{\min}Qjmin​ be the lowest performance achieved by any algorithm and Qjmax⁡Q_{j}^{\max}Qjmax​ the highest. By construction, on each task this gives relative performance: a value of 0.8 means the algorithm achieved 80% of the range between worst and best on that task. Averaging these relative scores gives an overall measure of performance independent of absolute scale differences between tasks. This strategy is very useful when different tasks have different difficulty levels or score ranges - it prevents a high-variance task from overwhelming the average. Essentially, each task contributes equally after normalization. A potential consideration is that it assumes a linear scale is meaningful (performance differences are equally important in proportion across tasks). Also, if there are ties at best or worst, or an outlier algorithm that performs much worse than others (making Qjmin⁡Q_{j}^{\min}Qjmin​ very low), the normalization can exaggerate differences. Still, Q-RM is a balanced way to aggregate when fairness across heterogeneous tasks is desiredar5iv.labs.arxiv.org.
6. Best Rank Count (R-B)
Intuition: Best Rank Count is like counting gold medals in the Olympics. It asks "on how many tasks was this algorithm the best performer?". This strategy tallies the number of first-place finishes an algorithm has. It highlights methods that win often, even if they might not be consistently top on every taskar5iv.labs.arxiv.org.
Technical details: Formally, best rank count R-B(i)R\text{-}B(i)R-B(i) is the number of tasks on which algorithm iii achieved rank 1 (the top rank). One can write R-B(i)=∑j=1m1{ri,j=1}R\text{-}B(i) = \sum_{j=1}^{m} \mathbf{1}\{r_{i,j} = 1\}R-B(i)=∑j=1m​1{ri,j​=1}, where 1\mathbf{1}1 is an indicator function that equals 1 if algorithm iii was first on task jjj, and 0 otherwise. An algorithm with a high R-B won a lot of "races." This measure is useful when we care about methods that can reach the top spot, which is common in contexts like AutoML or model selection where you might prefer a method that has the highest chance to be best on a new problem. However, R-B ignores how an algorithm performs when it isn't the best - it doesn't reward consistency or second-place finishes. For example, an algorithm that is first on 5 out of 10 tasks and mediocre on the rest will outrank an algorithm that is second on all 10 tasks (with no wins) under R-B. Thus, it's often interpreted alongside other metrics. If multiple algorithms tie for best on a task, typically each could be counted (or a fractional count could be assigned), but the simplest case assumes one clear winner per task. R-B is a simple count with no statistical assumptions, but it should be used when peak performance is the priority.
7. Worst Rank Count (R-W)
Intuition: Worst Rank Count is the flipside of R-B - it counts how often an algorithm came in last place on a taskar5iv.labs.arxiv.org. This asks "in how many tasks was this algorithm the worst performer?". It's like counting how many times a racer finished dead last in a series of races, which highlights how often a method utterly fails relative to others.
Technical details: Formally, worst rank count R-W(i)R\text{-}W(i)R-W(i) is ∑j=1m1{ri,j=N}\sum_{j=1}^{m} \mathbf{1}\{r_{i,j} = N\}∑j=1m​1{ri,j​=N}, where NNN is the total number of algorithms being compared (so rank NNN means last place). A smaller R-W is better - an ideal algorithm would never be worst on any task (R-W = 0). This metric is useful for assessing reliability or risk: an algorithm with a high R-W frequently falls to the bottom, indicating that in certain scenarios it performs very poorly. In contrast, an algorithm with R-W = 0 or very low rarely has catastrophic failures, even if it might not always be the best. R-W, like R-B, is a simple count and doesn't consider how close to last an algorithm was - it's an all-or-nothing per task. It also doesn't reward being second-worst versus middle-of-pack; it only penalizes outright worst performance. In practice, one might use R-W to avoid methods that have a tendency to crash on some tasks. No complex assumptions are needed for R-W, but if there are ties for last place, handling them (e.g. fractional counts) would be a consideration. Generally, R-W provides a cautionary viewpoint: it identifies algorithms that one might want to avoid if the worst-case scenario is a concern.
8. Threshold-Based Quality Comparison (Q-Thθ)
Intuition: Threshold-Based Quality Comparison measures how often an algorithm's performance is within a certain threshold of the best performance on a taskar5iv.labs.arxiv.org. Instead of only counting outright wins, we count a task as a "success" for an algorithm if it comes sufficiently close to the best. Ask "on how many tasks did this method achieve at least θ-fraction of the best score?". For example, with θ = 0.95, if the algorithm reaches 95% of the top score on a dataset, it counts as nearly a win for that dataset.

Often this sum is normalized by mmm (giving a fraction of tasks) for interpretation. This strategy generalizes the best-count (R-B): if θ = 1, Q-Th₁ is exactly the count of outright winsar5iv.labs.arxiv.org. For θ < 1, it also gives credit for near-wins. For instance, if θ = 0.9, an algorithm gets credit on a task as long as its performance is at least 90% of the winner's performance on that task. This rewards consistency at a high-performance level - an algorithm that frequently comes very close to the best will score well, even if it's rarely the absolute best. It addresses a drawback of R-B by not penalizing an algorithm that is second place by a tiny margin. The choice of θ is a user parameter: a stricter θ (close to 1) means only almost-tied performances count, while a looser θ (like 0.8) counts more cases as successes. In practice, θ might be set based on what is considered "practically the same" performance (for example, within a few percentage points of accuracy). Q-Thθ assumes the performance metric is on a ratio scale where such comparisons make sense. It's a flexible measure to identify algorithms that are almost always among the top performers rather than occasionally best with some bad showings.
9. Friedman-Nemenyi Test-Based Ranking (R-N<sub>emp-value</sub>)
Intuition: This strategy uses statistical tests to derive a ranking with confidence. Instead of purely aggregating numbers, it asks "are the differences in performance across tasks significant or could they be due to chance?". The process is: (1) check overall if any algorithm is better than others using a Friedman test, and (2) if yes, perform Nemenyi post-hoc tests to see which algorithms significantly outperform which. The final ranking is based on these significance findingsar5iv.labs.arxiv.org. In essence, algorithms are ranked such that if one is ranked above another, it's because there is statistical evidence it tends to perform better, not just a tiny average difference.
Technical details: The Friedman test is a non-parametric statistical test used when comparing multiple algorithms over multiple datasets (it's like an extension of the paired sign-rank test to more than two algorithms)ar5iv.labs.arxiv.org. We first rank the algorithms on each task (as for R-M) and compute the average rank for each algorithm. The Friedman test evaluates the null hypothesis that all algorithms have equal average ranks (no real performance difference). If this null is not rejected (p-value is high), the conclusion is that we cannot distinguish the algorithms - there would be no significant basis for an ordered ranking. If the Friedman test does reject the null (p below a chosen threshold, say 0.05), it implies at least one algorithm differs significantly. Then, pairwise Nemenyi post-hoc tests are applied: for each pair of algorithms, we check if their average ranks differ by more than the critical difference (a threshold that depends on number of algorithms, tasks, and the p-value)ar5iv.labs.arxiv.org. If algorithm A's average rank is significantly better than B's, we conclude A > B in the consensus ranking. Using all these pairwise comparisons, we construct a ranking: essentially, algorithms can be partially ordered such that no statistically significant contradictions occur. Often this results in groups of algorithms that are tied (no significant differences among them) and a partial order between groups. For example, one might say Algorithm1 ≈ Algorithm2 > Algorithm3 (meaning 1 and 2 not significantly different, but both better than 3). The notation R-N<sub>emp-value</sub> indicates we must choose a significance level (p-value) for the Nemenyi tests, which affects how strict the comparisons arear5iv.labs.arxiv.org. A common choice is 0.05. This approach is valuable when stability and confidence in the ranking are needed - it guards against over-interpreting random variations. The statistical assumptions here are that tasks are independent "blocks" and that the performance differences are consistent enough to be ranked (Friedman makes minimal distributional assumptions but does assume the rankings of algorithms on tasks are meaningful). Computationally, Friedman and Nemenyi tests are lightweight to compute even for many algorithms, but the interpretation must be done carefully. The result is a consensus ranking where any asserted ordering (A above B) is backed by statistical evidence of difference, making it a rigorous way to aggregate benchmark results beyond simple averaging.
10. Kemeny-Young Aggregation (R-Kem)
Intuition: Kemeny-Young aggregation finds a consensus ranking that is as close as possible to all the individual task rankings, treating the problem like a voting election among rankings. Imagine each dataset's ranking of algorithms is like a voter's preference list. The Kemeny-Young method asks "which single ranking of the algorithms would best agree with these myriad preferences overall?". It's akin to finding the ranking that minimizes the total "disagreement" (pairwise inversions) with all the per-dataset rankingsar5iv.labs.arxiv.org. This is often considered an optimal way to aggregate rankings because it uses all the information from each ordering.
Technical details: Kemeny-Young is based on an optimization principle. Consider any proposed consensus ranking of the algorithms. For each pair of algorithms (A, B), count how many datasets rank A above B versus B above A. A consensus ranking that puts A above B contributes a "disagreement" for each dataset that had B above A. The Kemeny score of a candidate ranking is the total number of pairwise disagreements with all the task rankings (or equivalently, the sum of distances like Kendall tau distance between the consensus and each task ranking). The Kemeny-Young aggregation finds the ranking that minimizes this total distancear5iv.labs.arxiv.org. Equivalently, it maximizes the total number of pairwise agreements with the input rankings. This method will satisfy a desirable property: if one algorithm beats another on a majority of tasks (pairwise), the consensus will rank it higher (it's a Condorcet-consistent method in social choice terms). Formally, finding the Kemeny optimal ranking is NP-hard in general, meaning it's computationally intensive for large numbers of algorithms. The description R-Kem indicates using this method; in practice the authors solved it by formulating a mixed-integer linear program (MILP) to find the optimal rankingar5iv.labs.arxiv.org. They included handling for ties and missing values in individual rankingsar5iv.labs.arxiv.org, since real benchmark data might have ties or un-evaluated cases. One should note that Kemeny aggregation can be much slower than simpler heuristics: the cited study reported it took minutes per aggregation with a state-of-the-art solverar5iv.labs.arxiv.org. Despite the computational cost, Kemeny-Young gives a very principled consensus. It makes no statistical assumptions; it purely seeks the ranking with minimal disagreements. This is useful when you want an unbiased aggregate that fully respects the order preferences from each task. For example, if you have 50 datasets each ranking 10 algorithms, Kemeny-Young will find the ranking of 10 that best reflects those 50 rankings as a whole. It's regarded as a gold-standard for rank aggregation because of this optimal consensus property. However, for many algorithms or complex scenarios, one must be mindful of the computational expense. In summary, R-Kem provides an exact, theory-backed consensus ranking at the cost of more computation, and it will coincide with our intuitive notion of the "fairest" aggregate ranking given all the tasks' resultsar5iv.labs.arxiv.org.
